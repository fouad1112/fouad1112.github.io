Contexte

Script fait seul pour comprendre comment on peut détecter une attaque sur un serveur Linux juste en lisant ses fichiers de log. L'idée m'est venue en installant Fail2ban sur Kinopio — je voulais savoir comment ça marche vraiment derrière. Juste Python de base, rien à installer.

Comment ça fonctionne
01

Lecture du fichier de log

Le script ouvre le fichier auth.log et lit chaque ligne une par une. C'est ce fichier que Linux utilise pour enregistrer toutes les tentatives de connexion.

02

Recherche des lignes suspectes

Pour chaque ligne, on regarde si elle contient Failed password ou Invalid user — ce sont les messages que Linux écrit quand quelqu'un rate une connexion SSH.

03

Récupération de l'adresse IP

Quand une ligne suspecte est trouvée, on cherche l'IP dedans avec re.search(). On compte combien de fois chaque IP a échoué, et si elle a essayé de se connecter en root.

04

Affichage et rapport

Les IP sont affichées du plus grand nombre d'échecs au plus petit. Celles avec 10 échecs ou plus sont marquées SUSPECT. Tout est sauvegardé dans rapport_sentinel.txt.

Extrait du code
log_sentinel.py
for ligne in lignes:
  if "Failed password" in ligne or "Invalid user" in ligne:
    ip = re.search(r"(\d{1,3}\.){3}\d{1,3}", ligne)
    if ip:
      ip = ip.group()
      if ip not in echecs:
        echecs[ip] = 0
      echecs[ip] += 1
      if "root" in ligne:
        tentatives_root.append(ip)

# affichage trie par nombre d'echecs
for ip, nb in sorted(echecs.items(), key=lambda x: x[1], reverse=True):
  ligne_rapport = ip + " | echecs : " + str(nb)
  if ip in tentatives_root: ligne_rapport += " | tentative root"
  if nb >= 10: ligne_rapport += " | SUSPECT"

Comment ça fonctionne
01

Saisie de la cible

L'utilisateur entre une adresse IP ou un nom de domaine. Le script démarre la collecte d'informations.

02

Résolution DNS

On récupère l'adresse IP du domaine avec socket.gethostbyname(), puis on essaie de faire l'inverse — retrouver le nom à partir de l'IP.

03

Ping, WHOIS, Nmap

Chaque commande est lancée avec subprocess.run() — c'est comme si on la tapait dans le terminal, mais depuis Python. Les résultats sont récupérés automatiquement.

Tout est écrit dans rapport_cyber.txt avec la date et le nom de la cible. Pratique pour garder une trace de ce qu'on a trouvé.

Extrait du code
outil_cyber.py
def executer_commande(cmd):
  try:
    resultat = subprocess.run(cmd, capture_output=True,
                         text=True, timeout=60)
    return resultat.stdout or resultat.stderr
  except: return "erreur pendant l'exécution"

# DNS resolution
ip = socket.gethostbyname(cible)

CTF Cryptographie Python
Déchiffrement RSA —
Module pair
Contexte

Script fait pour résoudre des challenges CTF avec du RSA mal implémenté. En CTF, on tombe parfois sur des clés RSA volontairement cassées pour qu'on puisse les casser. Ici le cas ciblé : quand le nombre N est pair, on peut retrouver la clé privée facilement et déchiffrer le message.

Comment ça fonctionne
01

Vérification de la vulnérabilité

RSA utilise deux nombres premiers p et q. Si N est pair, ça veut dire que p = 2 — on peut donc trouver q facilement en faisant N / 2.

02

Calcul de la clé privée

Une fois qu'on a p et q, on peut calculer φ(N) = q - 1, puis retrouver la clé privée d avec pow(e, -1, phi).

03

Déchiffrement du message

Avec la clé privée, on déchiffre le message avec pow(c, d, N) et on convertit le résultat en texte pour récupérer le flag.

Extrait du code
RSA.py
# Entrées utilisateur
N = int(input("Entre le module N : "), 0)
e = int(input("Entre l'exposant e : "), 0)
c = int(input("Entre le message chiffré c : "), 0)

# Vérification : N doit être pair
if N % 2 != 0: exit("Erreur : module impair")

# p=2, q=N/2, φ(N)=q-1
q = N // 2
phi = q - 1
d = pow(e, -1, phi) # clé privée

# Déchiffrement RSA
m = pow(c, d, N)
flag = m.to_bytes((m.bit_length() + 7) // 8, "big")
print(flag.decode()) # → CTF{...}
Ce que j'ai appris
Comprendre comment RSA fonctionne en pratique — pas juste en cours, mais en ayant un vrai résultat à la fin.
Identifier une erreur d'implémentation et comprendre pourquoi elle rend le chiffrement inutile.
Utiliser les calculs mathématiques de Python directement pour retrouver une clé privée.
Convertir un nombre en texte lisible pour récupérer le flag CTF.

Blue Team Infrastructure Projet BUT · 2024
Kinopio —
Sécurisation d'infrastructure
Contexte

Projet fait en équipe en BUT : on devait créer et mettre en ligne une plateforme d'enchères de A à Z. Mon rôle principal c'était de sécuriser le serveur — pas juste mettre le site en ligne, mais faire en sorte qu'il tienne face à de vraies attaques.

Ce que j'ai mis en place côté sécurité
01

Nginx — HTTPS uniquement

J'ai configuré Nginx pour qu'il soit le seul point d'entrée du serveur. Toutes les connexions HTTP sont redirigées vers HTTPS automatiquement. L'application Docker n'est pas exposée directement.

02

Fail2ban — Blocage des attaques par force brute

Fail2ban surveille les logs du serveur. Si une IP rate trop de fois sa connexion, elle est bloquée automatiquement. Ça protège contre les robots qui essaient des milliers de mots de passe.

03

Chiffrement SSL/TLS

Certificat SSL mis en place sur Nginx pour que tout ce qui passe entre le site et les visiteurs soit chiffré. Les mots de passe et données de paiement ne circulent pas en clair.

04

Sauvegardes — Règle 3-2-1

3 copies des données, sur 2 endroits différents, dont un hors du serveur principal. Comme ça si le serveur tombe, on ne perd rien.

05

Supervision des services

Mise en place d'un suivi en temps réel pour savoir si les services tournent bien et être alerté si quelque chose plante.

06

Docker + déploiement automatique

L'application tourne dans un conteneur Docker avec un utilisateur qui n'a pas les droits admin — si le conteneur est compromis, les dégâts sont limités. Le déploiement se fait automatiquement via GitHub Actions.

Extrait — Dockerfile (sécurité)
Dockerfile
# Build multi-stage — réduction de la surface d'attaque
FROM node:18-alpine AS base
# ...
FROM base AS runner

# Utilisateur non-root — principe du moindre privilège
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs
USER nextjs

# Port unique exposé
EXPOSE 3000
CMD HOSTNAME="0.0.0.0" node server.js
Ce que j'ai appris
Configurer Nginx en conditions réelles — pas juste en cours, avec un vrai serveur accessible depuis internet.
Comprendre comment Fail2ban fonctionne et pourquoi c'est indispensable dès qu'un serveur SSH est exposé.
Limiter les droits d'un conteneur Docker pour que si quelqu'un rentre dedans, il ne puisse pas faire grand chose.
Mettre en place des sauvegardes et comprendre pourquoi avoir tout au même endroit c'est risqué.
Travailler en équipe avec Git et livrer un projet fonctionnel de bout en bout.
